# TrackAI: как устроен трекинг траектории

Подробное описание пайплайна по коду (`backend/video_tracker`, `backend/main.py`). Это не «классический object tracking по ID на всём видео», а **визуальная одометрия (Visual Odometry)** по последовательности кадров с опциональной привязкой к человеку в кадре.

---

## 1. Что именно «трекается»

Система оценивает **относительное движение камеры** между соседними (по обработке) кадрами: сдвиг \((dx, dy)\) в локальной системе и поворот \(\Delta\theta\) (yaw). Эти приращения накапливаются в **мировую траекторию** \((x, y)\) и **курс** `heading_deg` (третья координата каждой точки).

Это **не** трекинг по GPS и не SLAM с картой помещения на этапе ядра — карта плана подключается **после** как постобработка, если пользователь передал контекст плана.

---

## 2. Конвейер от API до результата

Типичный путь (`main.py` → `processor.py`):

1. **Загрузка видео** на сервер.
2. **Опциональная стабилизация** (`stabilize_video`): FFmpeg-сглаживание тряски, чтобы матчи признаков и optical flow были стабильнее.
3. **Чтение кадров** в `FullFeatureProcessor.process_video`: кадры **масштабируются** по ширине до `target_width` (по умолчанию 900 px), чтобы скорость была предсказуемой.
4. На каждый обрабатываемый кадр вызывается **`EnhancedVisualOdometry.process_frame`** (или упрощённый пресет — см. ниже).
5. Результат: массив точек `trajectory`, повороты, статистика качества; при наличии плана — **`apply_map_postprocessing`** выравнивает путь по сетке проходимости.

В продакшен-конфиге API процессор создаётся с homography, Kalman, AKAZE, optical flow и ML ROI (`main.py`, создание `FullFeatureProcessor`).

---

## 3. Один кадр: подготовка изображения

В `EnhancedVisualOdometry.process_frame`:

- Кадр переводится в **grayscale**, к нему применяется **CLAHE** (выравнивание локального контраста) — лучше детектировать углы/линии.
- При первом кадре, если нет файла калибровки, **автооценка дисторсии** (`CameraCalibrator.estimate_distortion_auto`) может задать `camera_matrix` / `dist_coeffs` → затем **undistort**.
- Вычисляется **dt** между кадрами (`dt_sec`) из таймкода или `frame_skip`/FPS — для сглаживания и порогов «физичности» движения.

---

## 4. Зона интереса: человек vs вся сцена

### ML ROI (по умолчанию)

- Модель **YOLO-подобный детектор в ONNX** (`MLDetector`, класс «person»), multi-scale inference.
- Детекции проходят через **`SimpleTracker`**: сопоставление по **IoU + предсказанию центра по скорости** (упрощённый ByteTrack), выбирается **лучший трек**.
- Раз в `detect_interval` кадров обновляется bbox человека; по нему строится **маска** — дальше движение оценивается в основном по **фону вокруг/за человеком** (эго-движение камеры в коридоре), а не по текстуре одежды.

Если ONNX недоступен, включается fallback на **HOG people detector** (OpenCV).

### Режимы `person` и `ego`

`_resolve_tracking_mode` по геометрии bbox (размер, положение, соотношение сторон) голосует между:

- **`person`** — крупный человек по центру → маска из **переднего плана (MOG2) ∩ bbox**;
- **`ego`** — сцена «как от первого лица» → маска строится иначе (`_build_scene_mask`: обрезка краёв кадра, вырез зоны под человеком, фильтр по текстуре Laplacian).

От режима зависит, используется ли **sparse optical flow по всей сцене** (см. ниже).

---

## 5. Две независимые оценки движения

### A) Разреженный optical flow (режим `ego`)

- Точки **goodFeaturesToTrack** на сетке по кадру внутри маски.
- **Lucas–Kanade** пирамидально (`calcOpticalFlowPyrLK`), **forward–backward check** для отсева выбросов.
- По согласованным парам точек:
  - при наличии **калиброванной камеры** — сначала **`findEssentialMat` + `recoverPose`** → из матрицы вращения берётся **yaw** (поворот), **перенос из essential намеренно не используется как метрический шаг**;
  - иначе или как дополнение — **`estimateAffinePartial2D`** или **`findHomography`** (если `use_homography`) → \(tx, ty, d\theta\)).
- Инварианты качества: минимум точек, доля inlier после RANSAC, ограничение на величину сдвига относительно диагонали кадра.

### B) AKAZE + BF matching

- На полном grayscale детектируются **AKAZE** keypoints (с маской или без, если точек мало).
- **KNN-матчинг** с тестом Лоу (0.7×).
- Если матчей достаточно (`min_good_matches`), те же **essential / affine** пути, что и для sparse flow.

### Слияние (fusion)

- Если работают **и sparse, и features**: взвешенное среднее по **уверенностям** (`sparse_conf`, `feature_conf`).
- При **конфликте знака** поворота между двумя источниками — побеждает более уверенный; при равной неуверенности кадр **обнуляется** (учёт в `sign_conflict_frames` / `global_pose_failures`).
- Опционально **Farneback** dense flow в центральной области смешивается с переводом (вес зависит от того, был ли sparse).

Далее — **сглаживание** (`traj_smooth_alpha`, `turn_smooth_alpha`), при `use_kalman` — **Kalman 2D** по позиции, **мягкое уменьшение перевода при большом повороте**, затем **`_final_motion_gate`** отсекает нереалистичные скачки при низкой уверенности.

## 6. Из локального сдвига в глобальную траекторию

Интеграция **в плоскости**:

- Локальный вектор \((dx_s, dy_s)\) поворачивается на **текущий heading** (как у мобильного робота): `global_dx`, `global_dy` добавляются к последней точке.
- **Heading** обновляется добавлением сглаженного \(\Delta\theta\) (с учётом `yaw_sign` для согласования «лево/право» с реальностью при необходимости).

**Straight mode (коридор):** если нет активного поворота и последние кадры «прямые» (малый средний |Δyaw|), вызывается **`get_dominant_corridor_heading`**: Canny + Hough lines, выделение доминирующего направления стен/линий (Manhattan), при высокой уверенности heading **чуть подтягивается** к оси коридора — уменьшается дрейф на длинных прямых.

## 7. Детекция поворотов

Два канала (оба заполняются в процессе):

1. **`raw_turn_points` (`_detect_turns_from_heading`)**  
   - По истории `turn_step_history`: фиксируется **эпизод**, когда средний |dyaw| выше порога «начало», затем ниже порога «конец».  
   - Суммарный угол по эпизоду ≥ ~20° → событие поворота с типом left/right по знаку суммарного yaw.

2. **`trajectory_turn_points` (`_detect_enhanced_turns`)**  
   - На окне по **геометрии полилинии** (хорда между векторами сегментов) — угол между направлениями «вход» и «выход»; если больше адаптивного порога — отдельная точка поворота с источником `chord`.  
   - Сверка с последним «yaw»-поворотом: при несогласии знака/величины снижается confidence и помечается возможный конфликт (`turn_conflict_count`).

В API в итоге часто отдаются **`turn_points` = raw**, а для UI/карты могут использоваться уточнённые варианты после карты.

---

## 8. Масштаб и «метры»

- Пиксельные сдвиги умножаются на **`motion_scale`** и **`scale_factor`** (задаётся с фронта/запроса, по умолчанию в коде порядка 12.3 — калибровка под сценарий «примерные метры»).
- **Монокулярная VO не знает абсолютный масштаб** без внешней привязки; `scale_factor` — основной рычаг привязки к реальным размерам.
- При заданной ширине коридора в метрах возможна **дополнительная подстройка** `scale_factor` по прямым участкам (`ScaleEstimator` в ветке с `corridor_width_m`).

**Оценка пройденного пути** — сумма длин сегментов между соседними точками траектории в этих же условных единицах (`get_statistics` → `estimated_distance`).

---

## 9. Постобработка по плану этажа (не ядро трекинга)

Если переданы `floor_plan_data` / `drawn_plan` и опорные точки, **`apply_map_postprocessing`**:

- Строит **occupancy grid** по плану, **скелет** проходов;
- Масштабирует и переносит траекторию в координаты плана;
- **Подправляет путь** (граф, A* по сетке, сглаживание) — получается **`map_trajectory`** и **`map_turn_points`**.

Это уже **геометрическое согласование** с картой, а не «сырой» VO.

---

## 10. Параметры, влияющие на поведение (из API)

| Параметр | Назначение |
|----------|------------|
| `scale_factor` | Масштаб траектории в условных единицах («примерные метры»). |
| `stabilize` | Предобработка видео перед VO (FFmpeg). |
| `detect_interval` | Как часто пере-детектировать человека (ML/HOG). |
| `turn_vote_threshold` | Порог в конфиге VO (ограничение 1–5 в `EnhancedVisualOdometry`). |
| `use_ml_roi` | Включение ONNX-детектора и трекера bbox. |

---

## 11. Ограничения и типичные сбои

- **Одна камера** → масштаб по глубине не наблюдаем напрямую; ошибки накапливаются (дрейф).
- **Слабая текстура, блики, motion blur** → мало матчей, рост `ransac_failure_rate` / пустые кадры.
- **Конфликт sparse vs features** → обнуление кадра, дырки в траектории.
- **`degraded_flags`** в результате процессора помечают подозрительные прогоны (мало движения, много отказов RANSAC и т.д.).

## 13. Пресет `HighAccuracyVisualOdometry`

В `slam_wrapper.py` класс `HighAccuracyVisualOdometry` наследует `EnhancedVisualOdometry` с фиксированными флагами: homography, Kalman, ML ROI включены; при отсутствии «расширенных» опций в `FullFeatureProcessor` может использоваться этот пресет вместо ручной сборки VO.
